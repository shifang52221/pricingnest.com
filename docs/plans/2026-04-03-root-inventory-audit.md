# Root Inventory Audit

## Goal

Audit the repository root during the unified recovery release without deleting
anything inside the release batch.

## Audit method

- Listed all root-level files and directories in the release worktree.
- Checked tracked status for suspicious root items.
- Searched the codebase and docs for references to suspicious root items before
  classifying them.

## Root summary

- `docs/`
- `public/`

- `src/`
- `tests/`

### Generated local-only directories

These are not release-scope leftovers. They are local environment artifacts or
verification output:

| Item | Classification | Evidence | Later action |
| --- | --- | --- | --- |
| `.astro/` | `clearly unused` for repo cleanup, but expected local cache | Not tracked in git; generated by Astro during local work | Leave alone; do not include in cleanup batch unless clearing local cache intentionally |

| `node_modules/` | `clearly unused` for repo history, but required local dependency install | Not tracked in git; standard local dependency directory | Leave alone; never treat as release cleanup scope |

### Historical tracked audit artifacts

| Item | Classification | Evidence | Later action |
| --- | --- | --- | --- |
| `audits/` | `requires more confirmation` | Tracked in git, contains `batch-001` through `batch-036` plus top-level audit exports; no runtime/build/test references found in current source search | Review separately in a cleanup-only or archive-only batch; do not delete during recovery release |

## Detailed note on `audits/`

`audits/` is the only root-level item that currently looks like a real cleanup
candidate rather than normal project structure or local build state.

Findings:

- It is versioned in git, so deletion would be a repository-history decision,
  not a local cleanup.
- It contains a large set of historical crawl and issue exports.
- Current build, runtime, and test paths do not depend on it.
- Because it is tracked and historically useful, it should be reviewed as its
  own decision rather than mixed into a site-quality release.

Recommended follow-up for a later dedicated batch:

1. Decide whether `audits/` is archival evidence that should remain in-repo.
2. If not, move it to an external archive or a narrower historical location.
3. Only then remove or relocate it in a cleanup-only change.

## Conclusion

No root-level deletion should happen in the unified recovery release.

The only item worth a later cleanup decision is `audits/`, and even that should
be handled in a separate batch because it is tracked historical material rather
than disposable local output.
