# Pricing Tier Cluster Design

## Goal

Strengthen the retained "pricing tier design" topic cluster so Google and human readers both see a clearer editorial
center, cleaner subtopic roles, and a stronger internal progression from metric choice to tier structure.

## Scope

This wave focuses on one compact retained cluster:

- `pricing-tier-design`
- `usage-tier-breakpoints`
- `value-metric-selection`
- `pricing-metric-validation`

## Why This Cluster Matters

The site has already removed many weak inventory pages and upgraded the thinnest retained pages. The next structural
gain is no longer page count reduction alone. It is making the strongest retained topics behave like real hub-and-spoke
systems.

This cluster is a high-leverage one because it sits close to core commercial questions:

- what should be priced
- how tiers should be structured
- how metric choice and validation feed plan design
- where usage breakpoints should be placed

If these pages feel disconnected or uneven, the site looks broad but not tightly reasoned.

## Design Principles

- Strengthen editorial roles, not just length.
- Make one page the clear hub and the others clearer supporting pages.
- Remove overlap by clarifying what each page is supposed to answer.
- Improve next-step routing so readers move through a decision sequence, not a flat inventory.
- Avoid expanding this into a broad packaging rewrite.

## Cluster Roles

### Hub

- `pricing-tier-design`

This page should behave as the main cluster hub. It should clearly explain:

- when tier design becomes the core problem
- which adjacent pages answer which sub-questions
- how metric choice, plan roles, and tier thresholds relate

### Supporting retained guides

- `value-metric-selection`
  - role: choose the underlying value anchor before structure design
- `pricing-metric-validation`
  - role: test whether the chosen billable unit is publishable and stable
- `usage-tier-breakpoints`
  - role: define one specific tier-structure subproblem around threshold placement

## Upgrade Pattern

### `pricing-tier-design`

Make the hub role more explicit by adding a stronger cluster-navigation layer and clearer references to when readers
should move into metric selection, validation, or breakpoint design.

### `usage-tier-breakpoints`

This is currently the weakest page in the cluster and still reads like a short checklist page. Upgrade it into a real
supporting guide with:

- stronger editorial metadata
- clearer sections on when breakpoint design matters
- guidance on how to interpret threshold movement and bill-shock risk
- tighter links back to the hub and adjacent pages

### `value-metric-selection` and `pricing-metric-validation`

Keep them strong but sharpen their role boundaries:

- metric selection should answer "what outcome/unit should anchor pricing?"
- metric validation should answer "is this candidate unit strong enough to publish and operate?"

## Success Criteria

- The cluster reads like a real decision sequence instead of four separate content assets.
- `pricing-tier-design` is visibly the hub.
- `usage-tier-breakpoints` is no longer a thin checklist page.
- `value-metric-selection` and `pricing-metric-validation` have less role overlap.
- Build, governance, and audit checks remain clean.
